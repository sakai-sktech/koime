# koetype 設計判断ログ

> 形式: 何を決めたか / なぜ / 検討した代替案 / 受け入れたトレードオフ。
> 後続セッションがこのログだけで「なぜこうなっているか」を追えることが目標。

## DD-001: ゼロから書く（既存 OSS をフォークしない） — 2026-07-08

- **決定**: IME・録音・STT クライアントを自前実装。j3soon/whisper-to-input,
  FUTO Voice Input, woheller69/whisperIME は設計の参照のみ。
- **なぜ**: ライセンス継承の制約を負わない。コードベース全体を把握した状態で
  フェーズ2（AI 推敲・ストリーミング）に進める。バッファ方式 MVP なら自前でも規模が現実的。
- **代替案**: whisper-to-input のフォーク（動くベースから出発できるが、他人の設計の理解と
  改造範囲がゼロイチ実装と同等以上になると判断）。
- **トレードオフ**: 実績あるコードの安定性を捨てる。IME 特有の罠（フルスクリーンモード、
  InputConnection のライフサイクル）を自分で踏む。

## DD-002: バッファ方式（録音→停止→一括転写→挿入）で MVP — 2026-07-08

- **決定**: リアルタイムストリーミングではなく、タップで録音開始/停止し、
  停止時に音声ファイル全体を STT API に送る。
- **なぜ**: WebSocket + PCM ストリーミングはモバイルで安定させるのが難しく
  （ネットワーク断・バッテリー・部分確定テキストの修正処理）、MVP の検証対象
  「IME として成立するか」と直交する複雑さ。Typeless 自体もバッファ的な UX。
- **代替案**: OpenAI Realtime API でのストリーミング転写（フェーズ2候補として温存）。
- **トレードオフ**: 発話終了から挿入まで数秒の待ちが出る。長い発話は 25MB /
  API タイムアウトの制約を受ける（MVP では録音上限時間で防御）。

## DD-003: 録音は MediaRecorder → m4a（AAC） — 2026-07-08

- **決定**: `MediaRecorder`（MPEG_4 コンテナ + AAC, 16kHz mono）で cacheDir に録音。
- **なぜ**: AudioRecord + 自前 WAV 化より実装が一桁単純で、圧縮済みだから
  アップロードが軽い。OpenAI transcription API は m4a を受け付ける。
- **代替案**: AudioRecord + PCM/WAV（ストリーミング化やローカル whisper.cpp には
  こちらが必要になる — フェーズ2でこの決定は覆る可能性が高い、と明記しておく）。
- **トレードオフ**: 生 PCM が手元に残らないため、転写エンジンをローカルに
  差し替える際は録音層も触ることになる。

## DD-004: STT は SttEngine インターフェースで抽象化、初期実装は OpenAI — 2026-07-08

- **決定**: `SttEngine.transcribe(audioFile, languageHint): Result<String>` を境界に、
  OpenAI `/v1/audio/transcriptions`（multipart, OkHttp）を第一実装とする。
  モデル名は設定で切替（gpt-4o-transcribe / gpt-4o-mini-transcribe / whisper-1）。
- **なぜ**: クラウド API なら端末負荷ゼロ・多言語精度が既に高い・MVP が最短。
  抽象化しておけばローカル whisper.cpp / 他プロバイダを後から差せる。
- **代替案**: 最初からローカル whisper.cpp（NDK ビルド + モデル配布 + 端末負荷で MVP が重い）、
  最初から両対応（工数倍）。
- **トレードオフ**: オフラインで使えない。API 利用コストがユーザー持ち（自分の API キー方式）。

## DD-005: API キーは SharedPreferences(MODE_PRIVATE) に平文保存 — 2026-07-08

- **決定**: ユーザーが設定画面で入力した API キーはアプリ専用領域の
  SharedPreferences に保存。EncryptedSharedPreferences は使わない。
- **なぜ**: androidx.security-crypto は deprecated 方向で、鍵ストア絡みの
  デバイス依存クラッシュの温床。アプリ専用領域は非 root 端末では他アプリから読めない。
- **代替案**: EncryptedSharedPreferences（deprecated リスク）、Keystore 自前実装（過剰）。
- **トレードオフ**: root 化端末・adb backup 経路では平文が読める。個人利用の
  自分キー方式では許容。ログ・例外メッセージにキーを出さない規約で補強（CLAUDE.md）。

## DD-006: ツールチェーンはローカルキャッシュに合わせる — 2026-07-08

- **決定**: Gradle 8.2 / AGP 8.2.0 / Kotlin 1.9.20 / compileSdk 34 / minSdk 26 / targetSdk 34。
- **なぜ**: すべて ~/.gradle キャッシュ済みで、巨大ダウンロードなしにビルドが回る。
  AGP 8.2 は JDK 17（環境の Java と一致）+ compileSdk 34（SDK に platforms;android-34 あり）。
  minSdk 26 で MediaRecorder/IME の API が素直に使える。
- **代替案**: 最新（AGP 8.9+/compileSdk 36）— ダウンロードと互換検証に時間を食うだけで
  MVP に新 API の必要がない。
- **トレードオフ**: いずれ Play 提出するなら targetSdk 更新が必要（Play 要件は毎年上がる）。
  その時はツールチェーン一式を同時に上げる（CLAUDE.md に明記済み）。

## DD-007: UI は古典的 View（Compose 不使用）・依存最小 — 2026-07-08

- **決定**: キーボード UI は XML レイアウト + 通常 View。依存は
  kotlinx-coroutines と OkHttp のみ。JSON は org.json（Android 同梱）。
- **なぜ**: InputMethodService のキーボードビューは View 前提の API
  （`onCreateInputView`）で、Compose を IME に載せるのは lifecycle owner の
  手当てが必要な応用問題。設定画面程度に Compose を入れる価値がない。
- **代替案**: Compose（フェーズ2で設定画面だけ移行はあり得る）。
- **トレードオフ**: UI コードが冗長になるが、MVP の画面数は 2（キーボード・設定）。

## DD-008: 全画面エクストラクトモードを無効化 — 2026-07-08

- **決定**: `onEvaluateFullscreenMode()` を常に false。
- **なぜ**: エミュレータ実走（横長画面）で、IME 表示時にシステムが全画面エクストラクト
  モードに入り、アプリ本体が完全に隠れることを実測。音声入力では「どこに何を挿入するか」
  の文脈（アプリ画面）が見え続けることが UX の前提なので、隠す挙動は常に有害。
- **代替案**: デフォルト挙動のまま（横画面で全画面化）。
- **トレードオフ**: 極端に縦が狭い画面では入力欄がキーボードに隠れる可能性。
  実機で問題になったら再検討。

## DD-009: 例外は三層で受ける（入口正規化・境界Result化・scope最終捕捉） — 2026-07-09

- **決定**: (1) 設定保存時に API キーを正規化（全角英数→半角・空白除去）し、
  それでも非ASCIIが残れば保存を拒否して警告。(2) `SttEngine` 実装は
  CancellationException 以外の Throwable をすべて `Result.failure` に畳む。
  (3) IME サービスの CoroutineScope に CoroutineExceptionHandler を装着し、
  何が漏れてもプロセスを殺さずエラー表示に変換する。
- **なぜ**: 実機初テストで「変換中→即初期表示・挿入なし」を観測。エミュレータで
  再現に成功し、根本原因は API キー中の全角文字（0x30C6）で OkHttp の
  `header()` が IllegalArgumentException を即 throw → 未捕捉 → **IME プロセスごと死んで
  再起動**（再起動後の初期描画が「初期表示に戻った」ように見える）だった。
  ユーザーにはエラーが一切見えないため、無言クラッシュは症状の誤読を誘発する。
- **代替案**: engine の catch-all だけ（入口で防がないと「毎回エラーになるが理由が
  わからないキー」が保存され続ける）、入口検証だけ（境界契約が守られず将来の
  別例外で再発する）。三層は重複ではなく役割分担。
- **トレードオフ**: catch(Throwable) は本来のバグも握る方向の力学。Log.e("koetype")
  への出力を必須にして、logcat から必ず原因を追えることを条件に許容。

## DD-010: サービス化はスコープ外。BYOK を設計ポジションとして固定 — 2026-07-21

- **決定**: koetype 本体は今後も BYOK（ユーザー自身の API キー・自己責任）の
  個人ツールであり続ける。運営者付きサービス（キーを運営側に置くマネージド構成）は
  作らない。ただしフォークしてサービス化を考える人向けに、必要になるハードルの
  全体像を `docs/PRODUCTIZATION.md` として残す。
- **なぜ**: 「自分でハンドルを握って音声認識を操れること」がこのプロジェクトの
  価値の核。キーを自分で持つことは欠陥ではなく機能（コストの直接可視性・
  第三者経由なしのデータ経路・プロバイダ差し替えの自由）。サービス化は
  従量原価・悪用リスク・音声データの 3 つの運用責任を運営者が抱え込む構造転換で、
  OSS 個人ツールとしてのポジションと両立しない。
- **代替案**: 二層構成（無料 BYOK + 有料マネージド）— 技術的には `SttEngine`
  境界（DD-004）で共存可能だが、運用責任は同じだけ発生するため採らない。
  詳細な比較は PRODUCTIZATION.md §7。
- **トレードオフ**: 「API キーの取得」という初期障壁を越えられる人にしか届かない。
  それはターゲットの絞り込みとして受け入れる。

## DD-011: キー追加は段を増やさず既存の余白に置く（カーソルキー・全消しキー） — 2026-07-21

- **決定**: (1) カーソル移動キー ◀▶ をマイク行の左右デッドスペースに配置し、
  長押しリピート（400ms 猶予 → 50ms 間隔）を最初から実装。`KEYCODE_DPAD_LEFT/RIGHT` の
  `sendDownUpKeyEvents` で送出。(2) 「全消し」キーを状態表示行の右端（キーボード右上）に
  配置し、**直前に commitText した転写だけ**を取り消す。カーソル直前のテキストが
  転写文字列と一致する場合のみ `deleteSurroundingText` で削除（不一致なら何もせず
  状態表示で伝える）。入力欄が切り替わったら取り消し対象を破棄。
- **なぜ**: 転写結果は カーソル位置に入るため「挿入位置を直して話し直す」「外れた転写を
  丸ごと消す」が音声入力の基本動線。一方 IME の縦寸はアプリの可視領域と直接
  トレードオフ（DD-008 の前提）なので、キーを足しても高さは増やさない。
  マイク行の左右と状態表示行の右端は既存レイアウトの余白であり、高さコストゼロ。
  全消しの一致検証は「カーソル移動・手編集の後に誤って無関係のテキストを消す」
  事故を仕組みで防ぐため（backspace 連打より安全側に倒す）。
- **代替案**: キー段の追加（高さコストで却下）、全消し＝入力欄全体のクリア
  （IME が書いていないテキストまで消すのは越権で事故が大きい）、
  単発タップのみの矢印キー（10文字戻るのに10タップは実用にならない）。
- **トレードオフ**: 全消しは「転写直後」にしか効かない（スペース1つ手で入れた後は
  一致しなくなり無効）。安全側の割り切りとして受け入れ、状態表示で理由を伝える。

## DD-012: APIキー欄を Autofill Framework に開放する（偶発的摩擦の除去は意図したハードルの緩和ではない） — 2026-07-22

- **決定**: 設定画面の API キー入力欄の `android:importantForAutofill="no"` を
  廃し、`autofillHints` を付与して Android 標準の Autofill Framework に開放する。
  これによりユーザーが既に使っているパスワードマネージャ（1Password / Bitwarden /
  Google パスワードマネージャ / KeePassDX 等）がキー欄を直接フィルできる。
  ベンダー個別の SDK 連携・API 直結は今後も行わない。
- **なぜ**: koetype が意図的に置くハードルは「API キーを理解し自分で取得・
  管理できること」というリテラシーの選別であり、コピペ時の改行混入や長い
  キーの手入力はその選別に寄与しない**偶発的摩擦**。摩擦の除去はハードルの
  緩和ではなく、むしろ「鍵を自分の管理体制（マネージャ）で持つ人」への
  最適化＝BYOK ポジション（DD-010）の強化になる。クリップボードを経由
  しないため、改行混入（実害は DD-009 の normalizeApiKey で吸収済み）も
  他アプリからのクリップボード読み取りリスクも構造的に消える。
- **代替案**: (1) ベンダー API 直結 — 個人 vault へのモバイル公開 API は存在
  せず、サーバー製品（1Password Connect 等）は認証トークンの置き場という
  同じ問題が再帰する上、per-vendor 保守は分界点（DD-010）を跨ぐため却下。
  (2) GitHub Secrets / 認証アプリ連携 — 前者は write-only、後者は TOTP 専用で
  取得 API 自体が無く不成立。(3) 現状維持 — ユーザー自身のシークレット管理
  体制を標準機構の側から塞ぐ状態であり、塞ぐ積極的理由が記録にも無いため却下。
- **トレードオフ**: マネージャが「保存しますか」ダイアログを出すようになる
  （鍵をマネージャに保存させたい方向なので許容というより歓迎）。Autofill
  経由のデータはマネージャアプリを信頼境界に含めるが、それはユーザーが
  既に自分で選んで信頼している相手である。実機（マネージャ入り）での
  挙動確認が必要。

## DD-013: Google Play 非登録でも SDK 追随は続ける（期限付き義務ではなく保守衛生として） — 2026-07-22

- **決定**: Google Play には登録しない（配布は GitHub Releases の APK のみ）が、
  compileSdk / targetSdk とツールチェーン（Gradle / AGP / Kotlin）の追随は
  継続する。次にビルド構成へ手を入れるタイミングで compileSdk/targetSdk 36 +
  AGP + Gradle + Kotlin を一括更新し（DD-006 の「全部同時に」原則）、API 36
  エミュレータで IME 有効化・マイク権限・Autofill・録音中の画面遷移を一周
  検証する。以後は年 1 回程度の追随を目安とする。
- **なぜ**: Play の targetSdk 要件（2026-08-31 以降 API 36 必須等）は Play の
  審査ポリシーであり、非登録なら期限は消える（OS がインストールを拒むのは
  targetSdk 23 未満という大昔の水準のみ）。しかし期限が消えても実質的な
  理由が 3 つ残る。(1) 利用者の端末側 OS は勝手に新しくなり、古い targetSdk は
  互換モードという年々テストの手薄なパスで動くことになる。IME は互換挙動の
  影響を受けやすい部類（エッジツーエッジ強制・predictive back・マイク周りの
  プライバシー強化はいずれも targetSdk で挙動が切り替わる）で、検証 AVD が
  既に API 36 である現状はまさにこの状態。上げて壊れる箇所は放置しても
  いずれ OS 側変更で壊れるので、自分のタイミングで観測できるうちに上げる。
  (2) ツールチェーンの腐食 — 放置期間が長いほど「全部同時に」のジャンプ幅が
  広がって痛くなる。(3) 将来のローカル whisper.cpp 差し替え（DD-004 の布石）で
  ネイティブライブラリを積む際、新端末の 16KB ページサイズ対応に新しい
  NDK/AGP が必要になる。これは Play と無関係に端末側の現実として来る。
  なお SDK バージョン要件という論点自体、ChatGPT との壁打ち（非公開の
  原本リポジトリに収蔵）で指摘されるまで考慮外だった —
  「Play に出さない＝SDK を放置してよい」ではないことを明示的に残す。
- **代替案**: (1) 動く限り放置（凍結）— 依存が okhttp と coroutines だけの今なら
  凍結コストは低く見えるが、互換モードの経年劣化と将来ジャンプ幅の拡大を
  先送りしているだけで却下。(2) Play 期限（2026-08-31）に合わせて即時更新 —
  非登録なのでその期限に従う理由がなく、急ぐ必然もない。
- **トレードオフ**: 新 API を使う必要がないのに定期的な更新・再検証の工数を払う。
  年 1 回・一括・検証一周というリズムに固定することで、都度の判断コストと
  ジャンプ幅の両方を抑える割り切り。
