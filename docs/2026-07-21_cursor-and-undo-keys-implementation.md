# カーソル移動キー（◀▶ 長押しリピート）と全消しキーの実装

## Purpose & Context

OSS 公開に向けた機能拡張の壁打ち（同日 2026-07-21）から出た2機能の実装。
ユーザーの発案は (1) カーソル移動キーを「キー配置の上の段の左右」に置く、
(2) 自由割り当て可能な「付加キー」をさらにその上に置く、の2つだった。

壁打ちの結果、(1) は配置を変えて採用（新しい段ではなくマイク行の左右余白へ）、
(2) は見送り、代わりにアシスタント側から逆提案した「直前の転写を取り消すキー」を
「全消し」キーとして採用、という着地になった。配置方針の判断は DD-011 に記録済み。

前提となる文脈:
- 転写結果は `commitText` でカーソル位置に挿入されるため、「挿入位置を直してから
  話す」「外れた転写を丸ごと消す」が音声入力の基本動線であり、両機能はそれを塞ぐ
  ペインの解消。
- DD-008（全画面エクストラクト無効化）で確立した「アプリ画面が見え続けることが
  音声入力 UX の前提」が、今回の配置判断の上位制約として効いている。

## Implementation Decisions

### 1. カーソルキーは新しい段ではなくマイク行の左右デッドスペースに置く

ユーザーの当初案は「キー配置の上の段に左右1つずつ」。実装を確認したところ、
マイク行（`FrameLayout` 120dp、中央に 96dp のマイク）の左右が丸ごと余白だったため、
そこに 56×72dp で ◀▶ を置く案を提案し、ユーザーが「言おうとしてたことそのまま」と
即決で承認した。

- **理由**: IME の縦寸はアプリの可視領域と直接トレードオフ（DD-008 の前提）。
  段を足すたびにその前提を削る。既存余白なら高さコストゼロ。
  親指の到達性としても中央マイクの左右は一等地。
- **キーイベントは `sendDownUpKeyEvents(KEYCODE_DPAD_LEFT/RIGHT)`**:
  既存の backspace / enter と同じパターン。`InputConnection.setSelection` による
  カーソル位置直接操作ではなくキーイベント送出を選んだのは、既存コードとの一貫性による
  （両者の優劣は会話中では議論していない）。

### 2. 長押しリピートは「最初から」実装する

単発タップのみの矢印キーは「10文字戻るのに10タップ」を要求することになり
実用にならない、という判断で v1 から組み込んだ。品質の分水嶺はここにあると
壁打ち時点で明示している。

- 実装: `setOnTouchListener` で ACTION_DOWN 時に即1回発火 + 400ms 後から
  50ms 間隔で `mainHandler.postDelayed` によるリピート。UP/CANCEL で解除。
  ACTION_UP 時に `performClick()` を呼んでアクセシビリティ経路を残す。
- `InputConnection` 操作はメインスレッド、の不変条件（CLAUDE.md）と整合する形で
  `mainHandler`（メインルーパー）を使用。
- リピート速度 50ms は未検証の初期値。実機の体感で `REPEAT_INTERVAL_MS` を
  調整する前提で定数化してある。

### 3. 「全消し」= 入力欄全体のクリアではなく「直前の転写の取り消し」

ユーザーの発話は「今入れた文字列をすべて取り消せる（クリアできる）」。これを
「最後に `commitText` した転写文字列だけを消す」と解釈して実装した。
入力欄全体のクリア（全選択→削除）ではない。

- **理由**: IME が書いていないテキスト（ユーザーが元々打っていた文章）まで消すのは
  越権で、誤タップ時の事故が大きい。この割り切りは実装報告時にユーザーへ明示し、
  「欄ごと消したいなら変更は簡単」と選択肢も提示済み（本記録時点で異論なし）。
- **一致検証つき削除**: 消す前に `getTextBeforeCursor(last.length)` が記憶している
  転写文字列と一致するかを確認し、一致した場合のみ `deleteSurroundingText` で削除。
  カーソル移動・手編集の後は一致しなくなるため、無関係のテキストを消す事故が
  構造的に起きない。不一致時は `status_undo_unavailable` を状態表示に出す
  （無言で何も起きないのは DD-009 で学んだ「症状の誤読を誘発する」パターン）。
- **取り消し対象のライフサイクル**: `onStartInput(restarting=false)` で破棄
  （別の入力欄への移動後に前の欄の転写を「取り消せる」と誤認させない）。
  取り消し実行後も null に戻す（二度押しで二重削除しない）。

### 4. 全消しキーの配置は状態表示行の右端（キーボード右上）

ユーザー指定は「右上のあたりにドンと」。状態表示行（TextView 単独だった）を
横 LinearLayout 化し、statusText を weight=1、右端に 72×40dp のキーを置いた。
行の minHeight を 32dp→40dp に上げたので、キーボード全体で +8dp 程度の増加は
発生している（マイク行・下段は不変）。

### 棄却記録（Rejected Decisions）

- **付加キー（自由割り当てキー）— 今回スコープから外れた**: 壁打ちでは
  「割り当て対象の定義が設計のすべて」「v1 は定型テキスト挿入に絞るべき」と
  整理したが、ユーザーが今回選んだのはカーソルキーと全消しの2点のみ。
  明示的な棄却ではなく未着手。再開時は壁打ちログの3階層整理
  （定型テキスト / KeyEvent / アクション）から始められる。
- **新しいキー段の追加**: 高さコストで却下。IME が縦に伸びるほどアプリ本体が
  隠れ、「どこに挿入するか見えている」という音声入力の前提（DD-008）を削る。
  一度段を足すと後から減らす変更は既存ユーザーの手癖を壊すため、
  足す判断だけ一方通行に痛い。
- **全消し = 入力欄全体のクリア**: 上記の通り越権・事故リスクで棄却。
  ただし「本当に欄ごと消したい」ユースケースが実機で頻発するなら、
  長押しで全体クリア等の段階的な出し方はあり得る（完全否定ではない）。
- **単発タップのみの矢印キー**: リピートなしは「動くが使われない」機能に
  なりがち。修正距離が5文字を超えた瞬間にユーザーはアプリ側のタップに
  戻ってしまい、キーを付けた意味自体が消える。

## Key Learnings

- **クラウド実行環境（Claude Code on the Web）には Android SDK がない。**
  `local.properties` も gitignore 済みで存在せず、`assembleDebug` は実行不可。
  この環境でできるのは「既存パターンに忠実なコードを書く」ことまでで、
  コンパイル・実機検証は必ずローカルに残る。今後このリポジトリをクラウド環境で
  触るセッションは、最初からビルド検証をスコープ外として計画すべき。
- **既存レイアウトの余白は機能追加の一等地。** 「キーを足す=段を足す」と
  考えがちだが、FrameLayout 中央配置のマイクの左右、TextView 単独の状態表示行の
  右端、と探せば高さコストゼロの置き場が2つあった。
- **削除系の機能は「消す条件」より「消さない条件」の設計が本体。**
  一致検証・入力欄切替での破棄・実行後の null 化、と「誤って消さない」ための
  コードが削除実行の1行より多い。
