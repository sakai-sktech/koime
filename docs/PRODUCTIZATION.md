# koetype を「サービス」として提供する場合のハードル

> このドキュメントの読者は、koetype をフォークして（あるいは参考にして）
> **サービス事業として提供すること**を考えている人。
> koetype 本体は今後も BYOK（Bring Your Own Key）・自己責任の個人ツールであり続ける。
> その理由はこのドキュメントの最後に書く。

## 前提: koetype の現在の構造

```
[端末上の koetype] --音声 + ユーザー自身の OpenAI キー--> [OpenAI transcription API]
```

- API キーはユーザーが設定画面で入力し、端末のアプリ専用領域にだけ保存される（DD-005）
- 音声は端末の cacheDir に一時保存され、転写後に削除される。どこにも永続化されない
- 運営者は存在しない。コスト・リスク・データはすべてユーザー自身の手の中にある

これは「ツールの配布」であり、サービスではない。この構造のまま配布する限り、
作者側にはサーバー運用も、ユーザーデータの管理責任も、従量課金の原価も発生しない。

## サービス化とは何が変わることか

「ユーザーに API キーを入力させたくない。登録するだけで使えるようにしたい」——
サービス化の動機はほぼこれに尽きる。そして API キーを運営側に置いた瞬間、
構造は次のように変わる。

```
[端末上のアプリ] --音声 + 自前認証トークン--> [自前バックエンド] --音声 + 運営者の STT キー--> [STT プロバイダ]
```

このとき運営者が新たに抱えるものは 3 つ:

1. **従量課金の原価** — ユーザーが話すほど運営者に請求が来る
2. **悪用リスク** — 認証を突破されれば、そのバックエンドは「無料の Whisper API」として抜かれる
3. **ユーザーの音声データ** — 全ユーザーの発話が自分のサーバーを経由する

以下、この 3 つから派生するハードルを順に洗い出す。

## 1. STT プロキシバックエンド

構成はほぼ一択で、端末→自前バックエンド→STT プロバイダのプロキシ型になる。

- **STT プロバイダのキーはクライアントに一切出さない。** Secrets Manager
  （GCP Secret Manager / Cloudflare Workers Secrets 等）に置き、ローテーション可能にしておく。
  APK に埋め込んだキーは難読化しても必ず抽出される。「クライアントに置いたキーは公開したのと同じ」が前提。
- クライアント側の改修は小さい。`SttEngine`（DD-004）の実装を 1 つ足すだけで、
  IME・UI 層は無変更で済む。BYOK モードとマネージドモードの共存も同じ境界で実現できる。
- バッファ方式（DD-002）のうちは multipart を中継するだけなので、
  Cloud Run / Fly.io / Cloudflare Workers 程度の小さいインフラで足りる。
  ただし将来ストリーミング化（Realtime API / WebSocket）を見ているなら、
  **WebSocket を素直に張れる基盤か**を最初の選定条件に入れること
  （Workers は制約あり、Cloud Run / Fly.io は素直）。
- 音声が自サーバーを経由する分、レイテンシと転送コストが乗る。

## 2. 認証・ユーザー登録・課金

- 個人〜小規模チームなら auth の自作は最初から捨てる。**現実解は Firebase Auth か
  Supabase Auth**。Android なら Google サインイン一発が摩擦最小。
  バックエンドは ID トークン（JWT）を検証するだけの構造にする。
- Play ストアで課金するなら **Play Billing（サブスク）がほぼ強制**
  （デジタルサービスは外部決済を選びにくい）。手数料 15〜30% を原価計算に入れる。
- **購入検証は必ずサーバー側で行う。** Real-time Developer Notifications を受けて
  バックエンドが「このユーザーは有効なサブスクを持つ」を管理し、クォータと紐付ける。
  クライアントの自己申告で entitlement を決めない。
- 見落としがちな Play 要件: **アカウント作成があるアプリは、アプリ内からの
  アカウント削除機能が必須**（2024 年以降のポリシー）。auth を入れた瞬間に、
  削除フローとサーバー側のデータ削除 API も設計対象になる。

## 3. 悪用対策・クォータ

サービス化したバックエンドへの攻撃は「データを盗む」より先に
「転写 API をタダ乗りする」形で来る。IME を装った転写ファーム化は現実の悪用パターン。

- **クライアント側の制限は一切信用しない。** 録音上限などの防御はすべて
  サーバー側でも再検証する（音声長・ファイルサイズ・リクエストレート）。
  改造クライアントや curl 直叩きが前提。
- **ユーザー単位のクォータ（分/月）は最初から必須。** これがないと請求だけが青天井になる。
  クォータ管理はおまけ機能ではなく課金の根幹インフラ（→ §5 原価構造）。
- **Play Integrity API** で「正規クライアントからのリクエストか」を検証する。
  完璧な防御ではないが、悪用のコストを大きく上げる。
- 認証トークンは短命にし、漏洩時に個別失効できる設計にする。

## 4. プライバシー・法務

koetype にとってここが最も「重い」ハードル。IME はカテゴリとしてすでに
最重量級にセンシティブで（有効化時に OS が「このキーボードは入力内容をすべて
収集できる」警告を出す）、そこに「全ユーザーの音声が運営者のサーバーを経由する」が加わる。

- **「音声を残さない」設計をサーバー側にも延長する。** koetype の不変条件
  「cacheDir に置き転写後削除」をそのまま踏襲し、サーバーでは音声をメモリ/一時領域のみに置き、
  転写完了で即破棄。ログに音声・転写本文を残さない。
  これは実装するだけでなく**プライバシーポリシーに明文で謳う**。差別化要素にもなる。
- **STT プロバイダ側のデータの扱いを把握して明示する。** OpenAI API の場合、
  入力は学習に使われないが既定で 30 日保持される（zero data retention の申請余地あり）。
  規約は変わるので、サービス開始時点の一次情報を確認すること。
- **テレメトリを入れる場合も入力内容を絶対に含めない**規約を最初に固定する。
  クラッシュレポートに転写テキストが混入する事故は起きやすい。
- 法務面: 日本国内なら個人情報保護法（APPI）。海外配布するなら GDPR が視野に入る。
  音声は生体情報に片足かかっているため、Play の Data safety フォームの申告と
  実装の一致は雑にできない。プライバシーポリシーの整備は配布の前提条件。

## 5. 原価構造とプライシング

STT は従量原価なので、サブスク定額と食い合わせが悪い。ここを直視せずに始めると、
ヘビーユーザーに原価を食い尽くされる。

- 目安: gpt-4o-mini-transcribe で概ね $0.003/分（2026 年時点。必ず現行価格を確認）。
  1 日 30 分話すユーザーは月 900 分 ≒ $2.7 の原価。月額 500 円のサブスクなら
  ストア手数料を引いても成立するが、**使用量の分布は必ず偏り、上位数%が原価の大半を占める**。
- 対策は**フェアユースキャップを最初から仕様に入れる**こと
  （例: 月 1,000 分、超過は低速化・翌月まで停止・BYOK への誘導など）。
  後からキャップを入れるのは実質値上げ告知になり、最初から入れるより遥かに揉める。
- クォータの前提として、**ユーザー別の使用分数を記録する仕組み**が要る。
  ここで §4 との線引きが重要: 「転写した事実と分数は記録するが、内容は記録しない」。

## 6. ストア配布・運用

- **targetSdk の追従**: Play 提出には現行要件（毎年更新される）への追従が必要。
  koetype のツールチェーンはローカルキャッシュ最適化で固定している（DD-006）ため、
  Play 配布を目指すならツールチェーン一式の引き上げが初手になる。
- **署名は Play App Signing に移行**する（APK 直接配布用の keystore とは別管理になる）。
- **クラッシュ計測は必須**（Crashlytics 等）。koetype は「IME プロセスの無言クラッシュは
  症状の誤読を誘発する」ことを実機で経験している（DD-009）。他人の端末で起きる
  無言クラッシュは、観測手段がなければデバッグ不可能。ただし §4 の通り入力内容は絶対に送らない。
- リモート設定・強制アップデート機構・障害時のステータス告知など、
  「運営者がいるソフトウェア」としての運用装備が一式必要になる。

## 7. アーキテクチャより先に決めるべき戦略問題

技術的なハードルは上の通りだが、実はその手前に事業判断が 3 つある。

1. **BYOK モードを残すか、マネージド一本にするか。**
   - 二層構成（無料 = BYOK、有料 = マネージド）は `SttEngine` の差し替えだけで両立でき、
     無料層のインフラコストがゼロで、マネージド層が失敗しても製品が死なない退路になる。
   - マネージド一本は UX が最良（「API キー」という概念をユーザーに見せない —
     これが商用ボイスキーボードとの勝負条件）だが、トライアル原価も悪用対策も全部自分持ち。
2. **差別化はどこか。** 素の Whisper 中継だけなら Gboard 音声入力や
   Typeless / Wispr Flow と正面衝突する。AI 推敲（フィラー除去・整文）のような
   付加価値パイプはサーバー側に置く必然性があり、マネージド化とセットで初めて差別化になる。
   つまり「バックエンドを作る理由」は本当はキー管理ではなく付加価値機能かもしれない。
3. **誰の何のペインに課金してもらうか。** ここが決まらないとクォータ設計も
   プライシングも決められない。逆に決まれば §1〜6 の大半は機械的に決まる。

## なぜ koetype 本体はこれをやらないか

上のハードルは、どれも技術的には越えられる。越えないのは能力の問題ではなく**設計ポジションの選択**である。
また上記7つの章は、作者が業務で直面した現実的な制約等を知見としてまとめたものでもある。

koetype はOSSとしてリリースし「自分でハンドルを握って音声認識を操れること」に価値を置く。
API キーを自分で持つことは、この設計では機能と思想そのもの:

- コストが自分の OpenAI 請求として直接見える（中間マージンも不透明なクォータもない）
- 音声が経由するのは自分が契約したプロバイダだけ（運営者という第三者がいない）
- プロバイダもモデルも `SttEngine` の下で自分で差し替えられる（ローカル whisper.cpp を含む）

サービス化は、この「ユーザーの手の中にある」性質を運営者への信頼に置き換える行為だ。
それは正当なトレードオフであり、やりたい人はやればいい——そのための地図、ロードマップがこのドキュメントである。
ただし koetype 本体は、OSSとして置き換えない側に立ち続ける。
